Health Error Codes

Error Meaning Action
Actualization Has Not Run AP has a process called Actualization that is meant to run weekly after new core data is imported into the system (sales history, stock on hand, etc). This process incorporates actualized data into sales plans and makes any necessary adjustments. It is important that actual data is up-to-date in AP before making any subsequent changes to sales plans. Run actualization from the Job Manager. Contact Support if there are errors.
Actualized Sales Greater than Total The Total Sales metric is less than the Actualized Sales metric. Since Total Sales represents the lifetime of the item, adding Actualized Sales and Future Sales together, Total Sales should always be the greater number. This circumstance only occurs if there were some unexpected data condition (like extremely negative sales values in actualized data). There are many defenses in the application to prevent this from happening. Modify Total Sales to be greater than Actualized Sales, and Save. The Actualized Sales and Total Sales values can exactly match (and these values should match after the item has phased out), but there can be conflicts if entering in the exact Actualized Sales value into Total Sales and there are unrounded decimals not visible in the screen, so the value should be increased by 1.
Cluster Zero Site Count There are periods in the sales plan where there are no sites for the cluster set / cluster. This occurs because the cluster set snapshot active for the period contains no sites, or all of the sites in the cluster have been marked as closed. A sales, receipt, and inventory plan cannot exist where there are no sites. There are multiple actions to take to resolve this error. Reevaluate the cluster set snapshots over time, ensuring that all clusters have the right membership of sites. Reevaluate site status information, ensuring at least one site is open for each period of time during the life of the placeholder. If the cluster is truly meant to have no sites for some time (such as a cluster has been removed or added), then change the AP plan to exclude the time where the cluster doesn't exist. This can be done by changing phase dates of the placeholder to not include the periods where the cluster is empty, or un-assorting from the cluster altogether using the Cluster Details pane.
Data Expiration Has Not Run AP has a process called Data Expiration that is meant to run weekly after Actualization. This process prunes old data out of data tables to reclaim space in the database. How far out the data is pruned is governed by a system policy (which can also disable this feature altogether). It is important that expiration run weekly so that the expected amount of data is present in the sales plan. Run data expiration from the Job Manager. Contact Support if there are errors.
Insufficient Sales Plan    
Markdown Period Longer Than Expiration Period This error occurs if a very long markdown period was configured. In particular, there is an error if there is a Markdown set on an item with no Phase Out date, or if the Markdown period is longer than the Data Expiration period (a system policy). A Markdown phase this long is unexpected in AP data. Remove the Markdown date, or set the Markdown date closer to phase out. Markdown Dates should not be set for placeholders with no phase out date. In other words, the placeholder cannot be on markdown indefinitely.
Missing Receipt AUC A receipt period with planned receipt units does not have a corresponding Receipt AUC value. This can impact cost calculations throughout the sales plan. Add a Receipt AUC value. This can be done for all periods by modifying the value in main planning view, or by period in the Receipt Plan View.
Negative Sales Plan There are negative values in at least one period of the sales plan for this placeholder. Change sales values to a zero-valued or positive-valued number either at the aggregate level in main planning view or by period in Sales Plan View.
No Selling Periods The sales plan for this placeholder has no active selling periods. This would occur if the user manually toggled Selling Period to "off" for all periods, or the system set Selling Period to "off" for all periods if there are no sites in the cluster. Depending on the desired outcome for the user, several actions could be taken. If all periods were opted out in error, then the user should opt some periods back in as selling. If the periods are marked as non-selling due to having no sites in the cluster, then actions should be taken to correct cluster definitions (see the Cluster Zero Site Count error). If the item is meant to not be in phase yet, set the phase dates accordingly. If the placeholder should never be sold for the cluster, rather than opting out each period individually, the assortment should be removed using the Cluster Details pane.
Overstated Ticket Price    
Period Horizon Past Calendar For AP to function properly, a calendar extending into the future must be defined. This error surfaces if the defined calendar does not extend far enough into the future for placeholders with no phase out date. Placeholders with no phase out date have sales plans extended to the Period Horizon configured for the plan. If the calendar does not have periods defined at least that far out, then there is a conflict. The best thing to do is to run maintenance or imports to add more future periods into the calendar. In the short term, the placeholders with no phase out can have phase out set temporarily to the last defined period in the calendar until the calendar is extended.
