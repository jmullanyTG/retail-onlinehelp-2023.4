Aggregation levels and top down planning

Users can experience a slowdown when working with large volumes of data, whether it consists of many months, channels, product categories, or a combination of all these.

Aggregation levels limit the number of data rows the system must work with and display by pre-aggregating data below the selected level. In a top down planning environment, these can be utilized to great advantage by taking a tiered approach to working on plans:

For more information regarding the Product Agg Level parameter, see Plan parameters.

In the above example with the Product Agg Level set to Category, the Product Level Description pivot field shows Polo (5/5). This indicates there are 5 Brands beneath the Polo product category and the view is currently aggregating data from all of them.

By unchecking one of the Brands in the Product parameter dropdown and re-executing the view, the Product Level Description now reads Polo (4/5). This indicates data for only 4 of the 5 total levels is aggregated in the view.

By adopting a systematic approach, an optimal system performance can be maintained while working on progressively more granular levels of the plan. In the above example, the aggregation level has changed to Brand so work can be performed on the details of each Brand beneath the Polo category.

Locking functionality constrains changes made at the Brand level to the values planned at the Category level. For more information, see Lock & Spread.
