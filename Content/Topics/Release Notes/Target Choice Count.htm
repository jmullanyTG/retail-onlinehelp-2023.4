Configuring which clusters and cluster sets get more choices than others

To define a nested relationship for assortment choice counts, TG JE v2023.2 introduces functionality that lets users specify which clusters and cluster sets should always have more choices than others. The hierarchical configuration can be at the cluster level or at the cluster set level or both, depending on how users create segments and clusters.

Select the new Cluster Config button added to the toolbar to display the Target Item Master Group Cluster Configuration screen. Two new screen sections, Cluster Containment Configuration and Cluster Set Containment Configuration, added to this view allow users to specify which clusters and/or cluster sets should always have more choices than others.

In the above screen shot,

  • Cluster A will have more choices than Cluster B, which will have more choices than Cluster C.

  • Retail Cluster Set set will always have more choices than E-Com Cluster Set, which will have more choices than Outlet Cluster Set.

If clusters:

  • are unique across cluster sets (e.g. cluster E-Com only occurs in E-Com Cluster Set or cluster A only occurs in Retail Cluster Set), then it is sufficient to use the Cluster Containment Configuration section only.

  • are not hierarchical in nature (e.g. clusters are Warm Weather versus Cold Weather or East Coast versus West Coast), then use Cluster Set Containment Configuration only.

  • are repeated in cluster sets, and cluster sets have some hierarchical nature to them (e.g. Retail Cluster Set and Outlet Cluster Set both contain A, B, and C clusters), then use both sections as they can be used jointly.

Using cluster containment configurations in scenarios

To use a cluster containment configuration in a scenario, select the checkbox in the new Use Cluster Constraint field added to the segment configuration section.

This checkbox can be selected when Target Creation is accessed via the Assortment Planning module, or when in standalone Target Choice Count mode with Cluster as the specified Site Aggregation Level.

System policy Default: Use Cluster Constraint determines whether the Use Cluster Constraint field is checked or unchecked when new segments are created or generated.

Reviewing a constrained recommended choice count

A new metric labeled TY Constrained Recommended New Choices has been added to the Target Worksheet tab. This field shows the recommended choice count after it has been through cluster constraining as configured in the Target Item Master Group Cluster Configuration screen. If the constrained choice count differs from the recommended choice count, the text is formatted blue.

If a user changes the choices further (see TY ACT New Choices), the text is formatted blue if the value is increased and red if the value is decreased.

User-entered override choice count values are not included by the constrained logic. Cluster constraints are applied against system recommended choice counts. If using user-overrides, all clusters must be revised by the user as desired.

Using approximate stock quantities to assess recommended choices

Within the Target Worksheet and Target New Choices grids, recommended placeholders include Sales $ and Sales U targets. The stock needed to meet those Sales targets is not included in the Target Creation results due to Receipts being calculated later in the Assortment Planning process when other metrics such as Receipt eligible weeks, Presentation Minimums, Allow Receipts after Markdown, etc. are considered.

However, visibility to an approximate receipt or stock plan would greatly help planners decide whether to proceed with a choice recommendation. For example, planners may have an idea of the number of sizes needed per placeholder as a setup quantity. Depending on the store count within a Cluster Set/Cluster and the number of sizes needed for a presentation, the recommended Sales might not justify buying that choice, because the stock needed to setup is much greater than the Sales potential.

The target segment configuration section and the Target Worksheet now include the following two user editable metrics:

  • Target S/T %: Target Sales to Total Target Stock

  • Min Pres: The initial value is seeded from Hierarchy Defaults in AP mode. In standalone mode, the values do not seed. Only Hierarchy Defaults are used. Hierarchy Defaults by Cluster and Hierarchy Defaults by Time are not used.

Target S/T % and Min Pres can be defaulted at the segment level in the segment definitions widget, and both values can be overwritten at the cluster set / cluster level in the Target Worksheet.

Five new read-only Approximate Stock Unit metrics have been added to the Target Worksheet and Target New Choices grids. The metrics in the Target Worksheet are an aggregate for all new styles while in the Target New Choices grid, the metrics show values per choice.

To support differing strategic approaches to target stock levels at different organizations or product levels, there are two sets of metrics added to show approximate stock levels at stores at regular prices for new choices.

Firstly, approximate stock levels are calculated based solely on sell-through (ST%) targets without concern for maintaining minimum presentation throughout regular price selling periods. In this scenario, saleable stock is gradually depleted by sales throughout the regular price selling period, which may not be an issue where shoppers have lots of choices and are willing to substitute one choice for another. This approach helps avoid having excess inventory at the end of the regular price selling period for all styles. The metrics supporting this logic are:

  • Approx Stock Units (S/T %): TY Act New Sales Units / Target Sell Through %

  • Approx Stock Units By Site (S/T %): Approx Stock Units (S/T%) / TY Plan Site Count

Secondly, approximate stock levels are calculated with the requirements to have enough stock to meet sales and maintain a full minimum presentation throughout the regular price selling period. The metrics supporting this logic are:

  • Approx Stock Units (Min Pres): TY Act New Sales Units + (TY Plan Site Count * Min Pres)

  • Approx Stock Units By Site (Min Pres): Approx Stock Units (Min Pres) / TY Plan Site Count

  • Sales To Approx Stock Units (Min Pres): TY Act New Sales Units / Approx Stock Units (Min Pres)

An approximate stock target based on Sales plus Min Pres does not respond to a defined ST% target. The Sales To Approx Stock Units (Min Pres) metric is an Approximate ST% allowing for a comparison of the result of the (Min Pres) calculations to an ST%.

All 5 metrics are available for use as needed. If some of these metrics are not useful based on the strategy in place, those metrics can be hidden using the Fields selector.

Improved target planning

As part of an effort to expand the usability of the Target Choice Count functionality, four existing read-only metrics within the Target Worksheet are now editable. Although all four metrics are editable and can be selected as visible with the Field chooser, users can include 1 or more of these metrics within the Target Worksheet while not including the others.

The metrics are:

  • TY Act New Sales $ % +/- Comparison

  • TY Act New Sales U % +/- Comparison

Planning against +/- Comparison supports use cases where the target should be planned against the entire product hierarchy within the Comparison or set the +/- to LY actualized Sales within the cluster set cluster. For example, planning Jeans to LY actualized Sales +20%.

  • TY Act New Sales $ % +/- Comparison Active

  • TY Act New Sales U % +/- Comparison Active

Planning against +/- Comparison Active supports use cases where the target should be planned against only the Comparison Styles selected for inclusion in Comparison Active or set the +/- to LY actualized Sales for the styles designated for inclusion. For example, planning Core Jeans Styles +20%, rather than the entire hierarchy.

If one of the % +/- Comparison Active metrics is edited at the aggregate Cluster Set level, then all % +/- Comparison Active metrics for Clusters within that Cluster Set will change while holding the relationship between the those Cluster Set/Clusters. A likely more appropriate way to make this change would be at the Cluster Set/Cluster level. This would target specific areas, rather than move the entire Cluster Set. However, changes at the aggregate level are supported and the Cluster relationships will hold.

Maintaining AUR

Existing logic that AUR is always held is maintained. For example, modifying TY Act New Sales $ % +/- Comparison Active:

  • drives a new TY Act New Sales $ value

  • updates TY Act New Sales U, and

  • locks TY Act New AUR.

Improved control over one-off and poor performing variants

For TG JE v2023.2, the functionality associated with data cleansing in AP and Target Choice Count has been expanded with the addition of a new threshold and new fields that give users more control over one-off variants and habitual poor performers.

The new data cleansing threshold, Active Variant Total Minimum, lets users identify and remove consistently low sellers from the cleansing process. As with the other existing thresholds the default is set using a system policy. In this case, new system policy Cleanse: Active Variant Total Minimum (Module: Target Choice Count). The default is used when segments are created. However, users can change the value at the segment level.

The Active Variant Total Minimum functionality:

  • finds the total cleansed Active sales for all variants in the segment / cluster set / cluster

  • finds the average of these totals

  • multiplies the threshold to the average total to get the Minimum required

  • excludes any variants whose total is lower than this Minimum

Selecting the View Data Cleansing Report toolbar button displays the Target Data Cleansing report. The Target Data Cleansing Settings section displays the data cleansing parameters or thresholds, now including Active Variant Total Minimum, that are used for each segment. The background color of the threshold fields corresponds to data points that were cleansed or altered as a result of each.

For ease of use, all coloring rules used in the Target Data Cleansing report have been modified to purple to be more streamlined. This makes it easier to cross reference the background color of the threshold fields against the data points that were cleansed or altered as a result of each.

Choosing whether to rationalize placeholders

A new read-only checkbox labeled Carryover allows users to decide whether to include or exclude given placeholders from processing. If an item is to be carried over, but is near the end of its life and thus has a minimal sales plan in the current time period, then users can include the item so that Rationalization is informed by the item’s performance and recommends new placeholders to take its place. If the item is a carryover but it is expected to continue to sell with the same volume as last year, then the item can be excluded because its plan is already created and accounted for.

The method for population of the Carryover field depends on whether Target Creation is accessed through AP or standalone Target Choice Count:

  • In AP, Carryover is based on whether the placeholder’s phase dates intersect both the scenario Window of Time and the Comparison time period. For example, if the Comparison time period is over 2022 week 1 - week 10, and the scenario is over 2023 week 1 - week 10, an item selling from 2022 week 5 to 2023 week 15 will be considered carryover, but an item that phases out in 2022 week 30 is not.

  • In standalone Target Choice Count, Carryover is calculated with the same logic as in AP, but uses Inventory level phase dates instead of placeholder phase dates. If at least one of the variant’s inventories' phase dates intersect both Window of Time and Comparison time frames, then the variant is considered carryover.

Excluding placeholders from rationalization post data cleansing

A new editable field labeled Inclusion Type gives users the ability, even after data cleansing, to manually exclude some placeholders from consideration in the rationalization process. For example, a placeholder was a very unique item whose trends are not expected to be repeated in the future. The default setting is Full Process.

Setting Definition Possible Use Case
Full Process The variant will be included in full Rationalization processing, including in Data Cleansing. This is what the majority of items will fall into: non-carryover items that are a solid representation of the expected productivity and relationship between best to worst selling items.
Target Set Only The variant will be included in Data Cleansing to set Target values, but will be excluded from the Rationalization process. The item is a carryover item which should be included in Comparison Active data and target setting, but since it is already included in carryover, it should not be used for rationalization. The item was a strong seller last year due to intense societal trends, but it is not a repeatable scenario this year, so rationalization should not be informed by such a productive item.
Exclude The variant will not be included in any scenario processing. It will not influence Target or Recommended values. An item that should not have been selling in the cluster, so there is no desire to model it in the future.

Including items in data cleansing but excluding them from rationalization

Include Carryover In Rationalization?: Default is checked. If unchecked, then during the Calculate process, items flagged as Carryover are automatically set to Inclusion Type Target Set Only. They will be included in data cleansing but not in rationalization. This should be set if most carryover items in the Segment are long-life items that will continue to sell similarly to last year.

For more information regarding system policy Cleanse: Active Variant Total Minimum, refer to the System Policies section.

Using an additional attribute/group level to further align key strategies

Within Target Creation (and Target Choice Count), the addition of an optional Segment Split Type field that interacts with the Segment By Hierarchy functionality means the Segment By Hierarchy function can now apply down to an additional attribute/group level. This can be very helpful when creating targets which align to key strategies, such as: by brand, fashion, marketing program, or any product attribute as defined.

The Segment Split Type field is available as a dropdown when creating or modifying target scenarios with the dropdown comprised of:

  • Variant Hierarchy fields (but only those not a part of the configured Product Hierarchy)

  • Variant (for standalone TCC) or Placeholder (for AP) attribute fields - (only LOOKUP attribute types)

    When creating targets in AP, the entity in use is Placeholder (not Variant). There is no explicit assignment of hierarchy groups to Placeholders. Thus, if a Group type is chosen such as Brand, then the application will look up the Group value of the assigned variant. If there is no assigned variant, then the Lookup value is None.

The option (None) is intended to be used to clear previously selected options.

If an attribute is chosen, then segments are created on a cross section of hierarchy levels and the attribute selected. For example, if Brand is chosen, then the segments are the combinations of product hierarchy levels and brands.

Breaking Changes

ROS extrapolation updated to avoid crossing zero

Within Target Choice Count/Creation, previous functionality was to rationalize the comparison ROS values and make placeholder ROS recommendations to the point where additional recommendations are not mathematically justified. Beyond that point, if a user wished to add additional placeholders, the logic was to create a decreasing ROS slope to be applied against additional, user-requested recommendations. A function of the previous slope method was that the slope could reach or cross a zero (or negative) ROS.

The logic used to create the extended slope for ROS values to be applied against additional placeholders has been updated in such a way that it will never reach or cross zero. This allows users to create additional placeholders and receive suggested ROS values for each placeholder, regardless of the number of additional placeholders being requested. Creating a large amount of additional placeholders could result in very small ROS values for some placeholders. This is a function of a decreasing ROS slope.

Creating floor set updates for varying time frames

Target Creation (Assortment Planning or standalone Target Choice Count), provides a connection between MFP plan targets and an Assortment plan. This tool provides recommended choices, by cluster, using defined parameters. In previous functionality, choice recommendations were made using a defined Window of Time as the selling period for all recommendations. Within that structure (all choices being active during all selected periods), Target Scenarios had to be broken into multiple periods of time to get recommendations for a full season of choices.

To better support the work flow and processes of Assortment planning activities, Target Creation has been enhanced to allow users to define multiple floor set updates with their own windows of time within the selected overall period of time of a Target Scenario.

An example of this is a scenario with a 6-month window of time representing the Spring assortment, but further broken down into four floor set update periods corresponding to floor sets/product updates. The result being choice recommendations for each of those update periods.

To support the continuation of a choice selling across update periods, choices can be merged across update periods. For example, a choice recommendation from update period one, can be connected to a choice recommendation from update period two. This signifies that a single choice carries across periods of time, and one of the recommended choices is fulfilled in each update window.

The new functionality is made possible with the addition of a new widget labeled Segment Floor Set Updates on the right of the Edit Target Scenario view (whether creating a new or copying an existing scenario).

As floor set updates are segment-specific, segments must be created first. Therefore, users must click on segments for which updates are to be created and then use the Segment Floor Set Updates widget to configure the updates. Users have the option to assign the same update to all segments by selecting all segments or configure different updates by segment by clicking on one segment or smaller subsets of segments at a time.

Fields within the new widget include:

Field Description
Update Num Automatically generated and ordered by update period
Update Name (Optional) Provided for quick reference for major update initiatives such as Back to School or Christmas
Update Period (Required) Users can select any period within a scenario’s Window of Time. If the first update does NOT start at the beginning of the Window of Time, an update labeled Initial Set is automatically created. If that update is deleted, the following warning displays:

The first Floor Set Update does not include the first period(s) in the scenario Window of Time. These periods will not be represented in target setting.
Comparison Period (Optional) If not populated, then the chosen update period’s LY period automatically populates upon save.
Period Count, Comparison Period Count Automatically populates on save. The logic is that an update continues on until the next configured update. The last update goes to the end of the scenario’s Window of Time.

Therefore, if the Window of Time is wk1 through wk10, and 2 updates are configured for wk1 and wk5, then:

  • the first update goes from wk1 to wk4 for 4 periods, and

  • the second update goes from wk5 to wk10 for 6 periods.

Two other fields within the new widget, Selected Segment Count and Included Segment Count, give visualization to how many of the total selected segments fall into each particular update configuration.

Modifying updates or the Window of Time parameter

  • Changes to an update requires full recalculation and re-rationalization of the scenario. Data will be cleared from the Target Worksheet, Target New Choices, and Results Matrix grids.

  • Modifying a scenario’s Window of Time deletes all data AND update configurations. The following message displays with the option for users to proceed or cancel the change:

    Changing the Window of Time will delete the target spread data and segment update configurations. Proceed?

Merging placeholder choices

Building on the new ability to create floor set updates for varying time frames, planners can merge the choice recommendations for each of those update periods so that styles can continue across multiple updates before phasing out. While planners might define monthly updates which allow for the introduction of newness each month, it is very rare that the entire product offering will be replaced at each of these updates. It is more common that many choices will live through multiple updates, perhaps 3 or 4 monthly updates.

The choice count recommendations given are initially independent of each other. Meaning update 1 might recommend 6 choices, and update 2 might recommend 8 choices. This can be accomplished by phasing out all 6 choices and introducing 8 new choices OR by merging choices 1-6 from update 1 with choices 1-6 of update 2, resulting in a longer in-phase window for those choices, and allowing for the introduction of two new choices within update 2. The merging function supports this common use case.

After the merging is done as desired, the combination of the updates by period of time provides a choice count for each. The remaining choice recommendations can be used for newness.

To view and merge choices, select the new View Choices By Update toolbar button to display the new Target Choices By Update view.

Target Choices By Update view

The Target Choices By Update view, comprised of four widgets, lets users select which segments and locations to work with. Check the Sum Locations check box to sum locations. When checked, the data displayed collapses to show choices across all selected locations. When unchecked, specific values are broken out for the selected locations.

The toolbar has the following functionality:

  • Merge Choices: Attempts to merge a choice from one update to a choice from another update, so that the two (or more) choices become a single, long-living choice entity.

    • Choices from multiple updates can be selected at the same time.

    • Only one choice (per segment) can be chosen per update. For example, Choice A and Choice B from the same update cannot be merged together.

    • Choices must span a continuous range of time (choices from Update 1 and Update 3 cannot be merged unless a choice from Update 2 is also included).

    • When choices are merged, the Collection, Choice in collection assignment of the first update is retained.

  • Split Choice: Splits any merges done on a choice back into separate results. This operation can be done over several selected rows at once.

  • Auto-Merge: Attempts to automatically merge all choices for the selected rows' Collections and Choice In Collections. For example, Collection 1, Choice 2 is chosen, then any unmerged Collection 1, Choice 2 choices across all updates are found and merged together. This operation can be done over several selected rows at once (e.g. select Collection 1, Choice 1 - 5 and Collection 2, Choice 1 - 4).

  • Undo All: Returns the screen to its original state. In other words, it splits all merges.
